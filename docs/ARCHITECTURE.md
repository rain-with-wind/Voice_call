# 架构说明

## 设计目标

本项目的设计目标是把一个直接可运行的 TCP 语音通话脚本整理成结构清晰、方便维护和方便测试的工程。代码没有追求复杂的通信框架，而是围绕“点对点语音通话”这条主线拆分模块，让每个模块只负责一类事情。

主要考虑如下：

- 入口清晰：命令行解析和实际通话逻辑分离
- 配置集中：采样率、声道、端口、密码等参数由配置对象统一传递
- 音频独立：PyAudio 的初始化、设备选择、采集和播放集中在音频模块
- 通信独立：socket 连接、包格式、发送线程和接收线程由核心引擎管理
- 安全独立：认证和加密逻辑集中在安全模块，避免散落在通信流程中
- 可观察：运行时显示上传、下载、时长和麦克风音量，便于判断通话是否正常

## 总体结构

```text
用户命令
  |
  v
voice_call.py
  |
  v
voice_call_app.cli
  |
  v
VoiceCallConfig  --->  VoiceCall
                         |
        +----------------+----------------+
        |                |                |
      AudioIO      SecurityContext     CallStats
        |                |                |
   PyAudio 设备      认证 / 加密       状态显示
```

## 运行流程

### 服务端流程

1. 解析命令行参数，生成 `VoiceCallConfig`
2. 创建 `VoiceCall`
3. 创建 TCP socket 并绑定 `0.0.0.0:<port>`
4. 等待客户端连接
5. 如果设置了密码，进行认证握手
6. 如果启用加密并设置了密码，生成 salt 并配置 Fernet
7. 打开输入/输出音频流
8. 启动发送线程、接收线程和状态线程
9. 通话结束后关闭 socket 和音频设备

### 客户端流程

1. 解析命令行参数，生成 `VoiceCallConfig`
2. 创建 TCP socket
3. 连接服务端 `host:port`
4. 如果设置了密码，发送认证信息
5. 如果启用加密并设置了密码，接收服务端 salt 并派生 Fernet 密钥
6. 打开输入/输出音频流
7. 启动发送线程、接收线程和状态线程
8. 通话结束后释放资源

## 模块职责

### `voice_call.py`

项目入口文件。它只负责调用 `voice_call_app.cli.main()`，保留这个文件是为了让用户可以直接执行：

```powershell
python voice_call.py --mode server --port 5000
```

### `voice_call_app/cli.py`

负责命令行参数解析，包括运行模式、连接地址、端口、密码、加密开关、音频参数和设备选择。解析完成后创建配置对象，并根据 `--mode` 启动服务端或客户端。

### `voice_call_app/config.py`

定义 `VoiceCallConfig` 数据类。运行时需要的配置都从这里传入，避免多个模块各自读取命令行参数。

关键字段包括：

- `host`
- `port`
- `chunk`
- `rate`
- `channels`
- `password`
- `use_encryption`
- `input_device_index`
- `output_device_index`

其中 `frame_size` 用于判断 PCM 音频帧是否完整。

### `voice_call_app/audio.py`

负责音频相关工作：

- 初始化和释放 PyAudio
- 枚举系统音频设备
- 自动选择默认输入/输出设备
- 根据编号手动选择设备
- 打开麦克风输入流和扬声器输出流
- 读取音频块
- 播放音频块
- 计算麦克风音量

这一部分和操作系统环境关系最密切，尤其是 WSL 下可能出现 ALSA/JACK 日志，但只要设备可以打开并且数据速率正常，通话仍可使用。

### `voice_call_app/crypto.py`

负责轻量级安全能力：

- 使用 SHA-256 对共享密码生成哈希
- 通过 `AUTH:<hash>` 格式做密码认证
- 使用 PBKDF2 从密码和 salt 派生 Fernet 密钥
- 对音频负载进行加密和解密

当前设计适合课程项目、个人实验和可信网络中的内部使用，不适合直接作为高安全要求产品。

### `voice_call_app/engine.py`

项目核心模块，负责把网络、音频、安全和统计串起来：

- 服务端监听和客户端连接
- 认证与加密初始化
- 音频发送线程 `_record_and_send`
- 音频接收线程 `_receive_and_play`
- 实时状态线程 `_show_status`
- 固定长度接收 `_recv_exact`
- 资源关闭和运行统计输出

### `voice_call_app/stats.py`

负责记录发送字节数、接收字节数和最近一次麦克风音量，并把音量转换为终端进度条。

### `voice_call_app/console.py`

封装颜色输出，减少主逻辑里的格式化代码。

### `voice_call_app/windows.py`

处理 Windows 终端 UTF-8 输出，避免中文和符号显示异常。

### `tools/`

工具脚本主要用于排查环境：

- `wsl_audio_probe.py`：检查 WSL 音频设备
- `wsl_loopback_call_test.py`：测试本机回环通话流程
- `wsl_subprocess_socket_probe.py`：测试 WSL 下子进程和 socket 行为

## 数据封包格式

每个音频包由两部分组成：

```text
[4 字节网络字节序长度] [音频负载]
```

如果未启用加密，负载就是原始 PCM 数据；如果启用加密，负载就是 Fernet 加密后的数据。接收端先读取 4 字节长度，再按长度读取完整负载，最后解密并播放。

采用长度前缀的原因是 TCP 是字节流协议，不保留消息边界。如果不加长度字段，接收端无法判断一次 `recv` 得到的是完整音频块还是半个音频块。

## 线程模型

通话建立后会启动三个线程：

| 线程 | 职责 |
| --- | --- |
| 发送线程 | 从麦克风读取音频，按包格式发送 |
| 接收线程 | 从 socket 接收音频，解密后播放 |
| 状态线程 | 显示通话时间、上传、下载和音量 |

发送和接收分离后，双方可以同时说话和听到对方声音。退出时通过 `exit_event` 通知各线程停止。

## 安全模型

项目提供两层可选安全能力：

1. 认证：双方配置相同密码时，客户端发送密码哈希，服务端验证后才进入通话。
2. 加密：双方配置相同密码并启用 `--encrypt` 时，通过 PBKDF2 派生 Fernet 密钥，加密音频负载。

需要注意的限制：

- 命令行密码可能被系统进程列表看到
- 没有证书校验和身份体系
- 没有前向安全性
- 认证握手较简单，不适合公开网络中的高安全场景

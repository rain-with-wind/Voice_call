## 一、项目概述

本项目实现了一个基于 Python 的点对点语音通话工具。程序通过 TCP socket 建立连接，通过 PyAudio 采集和播放音频，并支持共享密码认证和可选加密。项目最终整理成了较完整的开源项目结构，包括主说明文档、使用说明、架构说明、测试说明、开发说明、贡献说明、安全说明和许可证。

我选择这个题目的原因是语音通话项目能同时涉及网络编程、音频处理、多线程、命令行交互和项目文档整理。相比只写一个算法脚本，这个项目更接近一个可以实际运行的小工具，也更适合练习工程化整理。

## 二、项目目标

项目主要目标如下：

1. 实现一对一实时语音通话。
2. 支持服务端和客户端两种运行模式。
3. 支持本机回环、局域网和 Windows / WSL 互通。
4. 支持麦克风和扬声器设备选择。
5. 支持共享密码认证。
6. 支持可选的 Fernet 加密。
7. 将代码整理成模块化结构，避免所有逻辑堆在一个文件中。
8. 补充项目运行、测试、开发、贡献和安全文档。

## 三、需求分析

## 四、总体设计

项目采用模块化设计，主要分为入口层、配置层、音频层、网络通信层、安全层和统计显示层。

```text
voice_call.py
  |
  v
cli.py 解析参数
  |
  v
VoiceCallConfig 保存配置
  |
  v
VoiceCall 组织通话流程
  |
  +-- AudioIO 负责音频采集和播放
  +-- SecurityContext 负责认证和加密
  +-- CallStats 负责状态统计
```

这样拆分之后，命令行参数、音频设备、加密逻辑和 socket 通信不会混在一起，后续修改某一部分时影响范围更小。

## 五、关键实现

### 1. TCP 连接

服务端通过 `socket.bind()` 和 `socket.listen()` 等待连接，客户端通过 `socket.connect()` 连接服务端。项目目前只支持一对一通话，服务端接受一个客户端后进入通话流程。

### 2. 音频采集和播放

音频部分使用 PyAudio。发送线程从麦克风读取 PCM 数据，接收线程把收到的数据写入扬声器输出流。默认音频参数是：

- 采样率：`48000`
- 声道：`1`
- 音频块大小：`1024`
- 格式：16-bit PCM

### 3. 数据封包

TCP 是字节流协议，不能保证一次接收刚好对应一次发送。因此每个音频包前面加 4 字节长度字段：

```text
[4 字节长度] [音频数据]
```

接收端先读取长度，再读取对应长度的数据，避免粘包和半包问题。

### 4. 多线程通话

通话建立后会启动三个线程：

- 发送线程：录音并发送
- 接收线程：接收并播放
- 状态线程：显示通话时长、速率和音量

这样双方可以同时说话和听到对方声音。

### 5. 密码认证

如果用户设置了 `--password`，客户端会发送密码哈希，服务端进行比对。认证通过后才继续进入通话流程，认证失败则断开连接。

### 6. 可选加密

如果双方都设置了 `--encrypt --password`，服务端生成随机 salt，双方用 PBKDF2 从共享密码派生 Fernet 密钥。发送前加密音频负载，接收后解密再播放。

## 七、测试过程

### 1. 编译检查

执行：

```powershell
python -m compileall voice_call.py voice_call_app tools
```

目的：检查 Python 文件是否存在语法错误或导入错误。

### 2. 命令行检查

执行：

```powershell
python voice_call.py --help
```

目的：确认命令行参数正常显示。

### 3. 音频设备检查

执行：

```powershell
python voice_call.py --list-devices
```

目的：确认程序可以列出输入和输出设备。

### 4. 本机回环测试

服务端：

```powershell
python voice_call.py --mode server --port 5000
```

客户端：

```powershell
python voice_call.py --mode client --host 127.0.0.1 --port 5000
```

通过标准：双方连接成功，上传和下载速率变化，麦克风音量条随声音变化。

### 5. 加密模式测试

服务端：

```powershell
python voice_call.py --mode server --port 5000 --encrypt --password test123
```

客户端：

```powershell
python voice_call.py --mode client --host 127.0.0.1 --port 5000 --encrypt --password test123
```

通过标准：双方认证通过，提示加密已启用，并能正常通话。

## 八、遇到的问题和解决方法

### 1. TCP 粘包和半包

一开始如果直接按固定次数 `recv`，可能会遇到数据不完整的问题。解决方法是给每个音频包加 4 字节长度前缀，并封装 `_recv_exact()`，确保接收完整数据。

### 2. 音频设备选择不稳定

不同系统的默认设备名称不一样，WSL 下也可能出现多个虚拟设备。解决方法是支持 `--list-devices`，并提供 `--input-device` 和 `--output-device` 手动选择。

### 3. 加密和普通模式兼容

加密只应该在双方都启用并且提供密码时生效。代码中通过 `SecurityContext` 集中处理，如果没有 Fernet 密钥，就直接返回原始数据。

### 4. 退出时资源释放

语音通话涉及 socket、PyAudio、输入流和输出流。项目通过 `exit_event` 通知线程退出，并在 `_cleanup()` 中统一释放音频资源，减少异常退出后的设备占用问题。

## 九、项目不足

当前项目仍然比较基础，主要不足包括：

- 不支持多人通话
- 不支持公网 NAT 穿透
- 没有图形界面
- 音频没有使用 Opus 等压缩编码
- 安全模型比较轻量，不适合高安全要求场景
- 自动化测试还不够完整，音频相关测试仍以手工验证为主

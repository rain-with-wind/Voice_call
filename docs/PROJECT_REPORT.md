# Voice Call CLI 项目报告

## 1. 项目概述

### 1.1 项目名称

Voice Call CLI：面向 WSL / Linux 虚拟机的语音通话与房间协调系统

### 1.2 项目背景

原始版本以单文件命令行脚本为主，能够完成最基础的服务端/客户端语音通信，但存在以下问题：

- 项目结构不清晰，不利于维护与扩展
- 缺少公网协调能力，用户只能手动输入 IP 和端口
- 前端交互缺失，操作门槛较高
- 仓库文档、许可证和工程化资料不完整

因此，本次工作围绕“课程作业可提交、工程结构更规范、适配 WSL/Linux 虚拟机、具备基础前后端分离思路”对项目进行了整理与扩展。

### 1.3 项目目标

- 将单文件脚本整理为多文件可维护项目
- 保留原有 `python voice_call.py` 兼容入口
- 增加公网房间注册与查询后端
- 增加便于操作的浏览器前端
- 使项目能够在 WSL / Linux 虚拟机中部署和验证
- 补全文档、注释、许可证与开源仓库基础材料

## 2. 功能需求与实现结果

### 2.1 直接语音通话

项目保留了经典的 TCP 双向语音通话模式：

- 服务端监听端口
- 客户端连接到指定主机与端口
- 双方通过麦克风采集与扬声器播放实现实时语音通信

### 2.2 公网房间协调

新增 `public_backend` 作为房间协调后端，支持：

- 创建房间
- 生成房间码
- 查询单个房间
- 更新房间在线状态
- 关闭房间

这样用户不必直接记忆 IP 地址，只需使用房间码或域名信息即可加入会话。

### 2.3 前端控制台

项目新增浏览器端控制台，保留核心功能：

- 创建房间
- 输入房间码加入房间
- 查看当前房间在线成员
- 展示当前房间的语音通话状态
- 关闭当前房间

前端不再暴露语音端口等底层连接细节，也移除了对普通用户不友好的实现术语。
同时，前端会在首次访问时生成并持久化设备 token，后端优先用该 token 识别成员，从而减少重复加入造成的成员重复显示问题。

## 3. 系统架构设计

## 3.1 总体架构

系统由三部分构成：

1. 语音传输层：负责真实 TCP 音频数据发送与接收
2. 房间协调后端：负责房间注册、查询与在线维护
3. 浏览器前端：负责以可视化方式操作后端接口

## 3.2 模块划分

### 语音 CLI 模块

- `voice_call.py`：兼容入口
- `voice_call_cli/cli.py`：参数解析与命令分发
- `voice_call_cli/engine.py`：音频流、Socket 与状态显示
- `voice_call_cli/public_commands.py`：基于后端的建房/加房流程
- `voice_call_cli/backend_client/api.py`：后端 API 客户端封装

### 后端模块

- `public_backend/app/__init__.py`：Flask 应用工厂
- `public_backend/app/routes/rooms.py`：房间相关路由
- `public_backend/app/room_registry.py`：房间数据持久化与状态维护

### 部署与测试模块

- `deploy/wsl/`：WSL 下的 CLI 环境准备与冒烟测试
- `deploy/public/`：后端部署脚本、服务配置与测试脚本

## 4. 关键实现说明

### 4.1 兼容性设计

虽然项目已拆分为多文件结构，但仍保留：

```bash
python voice_call.py --mode client --host 192.168.43.62 --port 5000
```

这保证了旧使用方式不会因工程重构而失效。

### 4.2 房间码机制

后端为每个房间生成 6 位短码，便于口头传播与手动输入。房间信息中包含：

- 房间名
- 公网主机名
- 端口
- 创建时间
- 更新时间
- 在线状态

### 4.3 前后端关系

本项目已经具备“前端页面 + 后端接口”的典型分层形式：

- 前端通过 `fetch` 调用 `/api/rooms/*`
- 后端以 JSON 响应房间信息
- 前端页面由后端静态托管，便于先在 WSL 中快速部署

严格意义上看，这属于“可独立前端逻辑 + HTTP 后端接口”的轻量分离方案，后续若需要，也可把前端单独部署到其他静态站点。

### 7.2 已完成验证内容

### 7.3 建议的实际通话验证

可通过以下方式进行两端联调：

服务端：

```bash
python3 voice_call.py --mode server --port 5000
```

客户端：

```bash
python3 voice_call.py --mode client --host 192.168.43.62 --port 5000
```

如果使用公网房间模式，则先启动后端，再通过页面或 CLI 创建房间。

## 8. 创新点与改进点

- 从单文件脚本演进为多文件项目
- 新增房间码与协调后端，降低连接门槛
- 新增前端控制台，减少纯命令行操作负担
- 保留旧命令兼容性，降低迁移成本
- 增补开源仓库所需文档与许可证

## 9. 不足与后续计划

当前版本仍存在以下不足：

- 语音流仍基于基础 TCP 传输，对复杂网络环境适配有限
- 浏览器前端不直接处理音视频
- 房间协调能力较轻量，当前不包含完整的用户体系
- 前端当前聚焦语音房间可视化，管理能力仍较基础

后续可继续扩展：

- 引入 WebRTC 以支持浏览器原生音视频
- 将前端独立构建为单独部署项目
- 增加自动化测试与 CI
